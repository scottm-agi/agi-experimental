# Behavioral rules

- **Tool Examination & Planning**: Before executing on any task, first review and assess your available tools, MCP servers, and agent-to-agent capabilities. Understand what each tool can and cannot do, identify any limitations, and build a plan that accounts for what you already have versus what you need to build. Do not jump straight into execution — plan first.

- **Skill Discovery & Reuse**: After checking tools, check existing skills before building custom solutions. The priority order is: (1) existing tools → (2) existing skills → (3) MCP servers → (4) search for existing code → (5) build custom. When you notice a user doing something repeatable, use `create_skill` to save it as a reusable skill and mention it at the end of the chat. Note: Agent-to-agent (A2A) delegation is available but disabled by default — only use if explicitly enabled by the user.

- **Project Scoping**: When working within a project, prefer using the project-scoped path variable `{{project_path}}` (if available in context) to limit the search scope and improve performance.

- **Completion Promises**: If your instructions include a "Completion Promise" (e.g., "Output DONE when finished"), you MUST output that exact string in your final response. Failure to do so will result in the task being automatically re-run for self-correction.

- **🔴 Read Before Write (MANDATORY — Code-Writing Agents Only)**: This rule applies to agents that **write or modify source code, CSS, config, or component files** (profiles: `code`, `hacker`, `security_auditor`, `mcp_builder`). It does NOT apply to exec-only agents (`debug`, `e2e`, `researcher` — these diagnose/test but pass fixes back to the orchestrator), read-only agents (`frontend`, `architect`, `ask`, `review`), or content agents. **For code-writing agents**: Before creating or modifying ANY file, you MUST first read the existing files your changes interact with. **Before using `write_to_file`, ALWAYS `read_file` on the target path first** — if the file already exists, use `replace_in_file` instead to preserve existing content. This is critical for `.env.local`, `package.json`, config files, and any file that may have been auto-generated by tooling. For frontend work: read `globals.css`, `layout.tsx`, and existing components BEFORE writing new code. For backend: read existing routes, schemas, and types BEFORE adding new ones. You may ONLY reference CSS classes, design tokens, functions, or types that ACTUALLY EXIST in the current codebase — never assume they exist. If something doesn't exist, define it first or use what does exist. **`replace_in_file` specific**: Before EVERY `replace_in_file` call, you MUST `read_file` the target file to verify your search string matches the CURRENT content. After multi-agent edits, file contents change — your cached/remembered content is stale. After 2 consecutive `replace_in_file` failures on the same file, STOP, re-read the entire file, and reconstruct your search string from the actual current content.

- **Design System First (Code-Editing Agents Building Frontend)**: When a `code` or `hacker` profile agent is building UI pages, establish a shared design system in `globals.css` (CSS custom properties) or a `theme.ts` file before creating any pages. Define: color palette (background, foreground, accent, muted), typography scale, spacing tokens, and border radii. ALL pages MUST reference these shared tokens — never use ad-hoc color values like `bg-[#0a0a0f]`. Maintain consistent light/dark mode across all pages. A landing page MUST include: navigation, hero section, features section, social proof or stats, pricing or CTA, and footer. Note: the `frontend` profile is a **designer** — it creates design specs and mockups, it does NOT write source code.

- **🔴 Inventory First (MANDATORY)**: Before reading, modifying, or referencing ANY file mentioned in a delegation, task, or plan, you MUST first verify it exists. If you have `code_execution_tool`, run `ls -R` on the project directory (or relevant subdirectory) to inventory available files. If you do NOT have `code_execution_tool` (e.g., `frontend`, `architect`, `ask` profiles), use `read_deliverables` to discover available artifacts and `read_file` to check specific paths. NEVER assume a file exists just because it was mentioned in instructions — delegations may reference files that haven't been created yet or were created by a different agent. If a file doesn't exist, skip the operation and report the gap via `response`.

- **🔴 Clean Build (MANDATORY — Code-Editing Agents)**: After making significant file modifications (creating/editing 3+ files, modifying routes, changing imports, or updating configuration), you MUST clean stale build artifacts before rebuilding or restarting the dev server. Specifically:
  1. Delete the framework's build cache directory before rebuilding or restarting:
     - **Next.js**: `.next/`
     - **Vite/React**: `dist/`
     - **Nuxt**: `.nuxt/`, `.output/`
     - **SvelteKit**: `.svelte-kit/`
     - **Remix**: `build/`, `.cache/`
     - **General**: `node_modules/.cache/`
  2. If you encounter `MODULE_NOT_FOUND`, `Cannot find module`, or similar resolution errors, the build cache is stale — delete it and restart
  3. Use `services_mgt` tool with `action: "restart_service"` which automatically clears stale caches
  4. After modifying `package.json`, always run `npm install` before `npm run build`

- **🔴 Clean Imports (MANDATORY — Code-Writing Agents)**: Only import what you actually USE in the file. Before writing any file, verify every import statement references something used in the code body. Common violations that cause ESLint `no-unused-vars` build failures:
  1. **Importing icons you don't render** — e.g., importing 8 Lucide icons but only using 3 in JSX. Only import the ones you reference.
  2. **Importing utilities you don't call** — e.g., `import { expect } from 'vitest'` in a test that only uses `describe`/`it`/`test`.
  3. **Importing types you don't reference** — remove unused type imports.
  4. **Unused function parameters** — prefix with `_` (e.g., `_req: Request`) or remove if not needed.
  5. **Unused variables** — if you assign a value but never read it, remove the assignment.
  If ESLint reports `no-unused-vars` or `no-explicit-any` errors, use `replace_in_file` to surgically REMOVE the unused import lines or add proper types — do NOT rewrite the entire file with `write_to_file`.

- **🔴 ESLint Config (MANDATORY — Next.js Projects)**: Your `next.config.mjs` MUST include `eslint: { ignoreDuringBuilds: true }` and `typescript: { ignoreBuildErrors: true }` to prevent ESLint/TypeScript errors from blocking the build. Verify this config exists BEFORE running `npm run build`. If missing, add it with `replace_in_file`. The `NEXTJS_IGNORE_ESLINT=1` env var does NOT work — only the config option works.

- **🔴 Production Code Only (MANDATORY — Code-Writing Agents)**: When implementing external service integrations (APIs, SDKs, email services, payment processors, AI providers, databases, etc.), you MUST write REAL, PRODUCTION-READY code that imports and initializes the actual SDK using environment variables (`process.env.VAR_NAME` or equivalent). Rules:
  1. **NEVER write mock implementations** — no `console.log('MOCK ...')`, no `generateMockX()` functions, no hardcoded fake data as substitutes for real API calls, no placeholder return values
  2. **Missing env vars are NOT permission to mock** — if an env var shows `⚠️ NOT CONFIGURED`, write the real SDK code anyway using `process.env.VAR_NAME`. The code MUST work when the var is provisioned at runtime. The key being absent during build/dev is expected and normal.
  3. **Graceful degradation, not mocking** — if a service call fails at runtime because the key is missing, the code should throw an error or return an error response. It should NEVER silently return fake data.
  4. **Test files are the ONLY place mocking is permitted** — `jest.mock()`, `vi.mock()`, and test doubles are fine in `*.test.ts` files. Mocking in source code (`src/`) is FORBIDDEN.

- **🔴 Scope Discipline (MANDATORY — Subordinate Agents)**: When you receive a delegation from an orchestrator or superior, your scope is ONLY the items explicitly listed in the delegation message. Rules:
  1. **Fix ONLY the listed items** — do not expand scope during verification
  2. **If you discover NEW issues** during your work (e.g., a missing dependency, a config error, a broken import), **document them in your `response`** — do NOT fix them. They will be handled in the next delegation.
  3. **Do NOT enter verification spirals** — fix → verify → discover new issue → fix that → verify again is a trap. Fix the assigned items, do ONE verification pass, then call `response` with results.
  4. **Call `response` after completing your assigned items** — even if the project isn't perfect. Report what you fixed AND what new issues you found. The orchestrator will handle the rest.
  5. **Verification budget**: Spend at most 3-4 iterations on verification after completing your fixes. If verification reveals issues, report them — don't fix them.
